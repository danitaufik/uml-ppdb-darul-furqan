Section 1

Use Case Diagram

Actors, grouped use cases, dan relasi «include» / «extend»

1.1 Actors

Actor Tipe Deskripsi
Guest Primary Pengunjung tanpa akun; mengakses halaman publik sekolah (landing, profil, berita, kontak, info PPDB) serta melakukan registrasi akun.
Calon Siswa Primary Pendaftar teregistrasi yang menjalani seluruh alur PPDB: verifikasi email, pengisian formulir, submit, pemantauan status, dan pengelolaan draft.
Orang Tua Primary Wali/orang tua yang mengisi data keluarga pada formulir PPDB dan memantau status penerimaan anak.
Admin PPDB Primary Operator sekolah yang memverifikasi atau menolak berkas pendaftaran, mengelola periode PPDB, dan mengekspor data.
Super Admin Primary Pengelola sistem penuh; memiliki semua hak Admin PPDB ditambah pengelolaan berita, profil sekolah, dan akun pengguna.
Email Service Secondary Sistem eksternal pengiriman email otomatis (verifikasi akun, notifikasi hasil verifikasi, dan pemberitahuan penolakan).

Activity Flow — PPDB

Diagram aktivitas alur proses PPDB dengan swimlane: User | System | Admin

2.1 Keterangan Swimlane

Swimlane Aktor Tanggung Jawab
User Guest, Calon Siswa, Orang Tua Interaksi langsung: melihat info, registrasi, login, mengisi formulir, submit, memantau status.
System Aplikasi PPDB (otomatis) Validasi data, pengiriman email, pemeriksaan periode aktif, pembaruan status pendaftaran.
Admin Admin PPDB, Super Admin Menerima notifikasi masuk, memeriksa berkas secara manual, dan membuat keputusan terima/tolak.

Sequence Diagram

Interaksi antar lifeline: User · Web App · Database · Email Service · Admin

3.1 Notasi Diagram

Diagram berikut menggunakan notasi UML standar: panah solid (→) untuk request/call, panah putus-putus (⇢) untuk return, batang aktivasi (activation bar) pada lifeline yang sedang memproses, dan kotak alt sebagai combined fragment untuk kondisi alternatif.

3.2 Sequence Diagram

3.3 Keterangan Notasi

Notasi Keterangan
→ solid Pesan request atau call — satu objek memanggil operasi pada objek lain.
⇢ dashed Pesan return — nilai balik setelah operasi selesai; biasanya berlabel :value.
Activation bar Kotak tipis di atas lifeline — menunjukkan periode objek aktif memproses.
sd <NamaDiagram> Sequence diagram frame — membingkai sebuah skenario lengkap.
alt Combined fragment tipe alternatif — garis putus-putus memisahkan kondisi; hanya satu yang dieksekusi.
[guard condition] Kondisi boolean yang menentukan cabang alt yang aktif, contoh: [credentials valid], [period active], [decision = accepted].

Class Diagram

Struktur class, atribut, method, enum, dan relasi antar class — Sistem Informasi PPDB

4.1 Class Diagram

4.2 Ringkasan Relasi

No. Dari Ke Tipe Kardinalitas Keterangan
1 PPDBPeriod PPDBApplication Aggregation ◇ 1 — 0..* Satu periode PPDB dapat memiliki banyak pendaftaran; pendaftaran tanpa periode tidak dapat ada.
2 User PPDBApplication Association 1 — 0..1 Satu user dapat memiliki paling banyak satu pendaftaran aktif per periode.
3 PPDBApplication StudentData Composition ◆ 1 — 1 Data siswa merupakan bagian integral formulir; tidak dapat berdiri sendiri tanpa pendaftaran.
4 PPDBApplication ParentData Composition ◆ 1 — 1 Data orang tua/wali adalah bagian integral dari formulir pendaftaran.
5 PPDBApplication Document Composition ◆ 1 — 1..* Pendaftaran wajib memiliki minimal satu dokumen yang diunggah.
6 User Notification Association 1 — 0..* Satu user dapat menerima banyak notifikasi selama siklus PPDB.
7 PPDBApplication Notification Dependency «triggers» — Perubahan status pendaftaran (submit/verified/accepted/rejected) memicu pembuatan notifikasi.
8 Admin User Generalization ▷ — Admin merupakan spesialisasi User dengan role=ADMIN; mewarisi seluruh atribut dan metode User.
9 SuperAdmin User Generalization ▷ — SuperAdmin merupakan spesialisasi User dengan role=SUPER_ADMIN; memiliki hak akses penuh sistem.
10 SuperAdmin SchoolProfile Dependency «manages» — SuperAdmin memiliki hak untuk membuat, mengedit, dan mempublikasikan konten profil sekolah.
11 SuperAdmin News Dependency «manages» — SuperAdmin memiliki hak untuk membuat, mengedit, dan mempublikasikan berita sekolah.
12 PPDBApplication ApplicationStatus Dependency «use» — PPDBApplication menggunakan enum ApplicationStatus untuk atribut status.
13 User UserRole Dependency «use» — User menggunakan enum UserRole untuk atribut role.
14 PPDBPeriod PeriodStatus Dependency «use» — PPDBPeriod menggunakan enum PeriodStatus untuk atribut status.

4.3 Referensi Enum

Enum Digunakan oleh Nilai (State Transitions)
ApplicationStatus PPDBApplication.status DRAFT → SUBMITTED → VERIFIED → ACCEPTED / REJECTED
UserRole User.role GUEST · APPLICANT · ADMIN · SUPER_ADMIN (assigned at register / promoted by SuperAdmin)
PeriodStatus PPDBPeriod.status UPCOMING → ACTIVE → CLOSED (controlled by Admin/SuperAdmin)
